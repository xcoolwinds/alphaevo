# Strategy Hub 规划

本文档回答 2 个问题：

1. AlphaEvo 是否有必要做一个社区化 Strategy Hub？
2. 如果要做，应该怎么做，需要哪些准备？

## 结论

- 有必要做，但不应该把它当成当前阶段的第一个优先级。
- 在 AlphaEvo 已经具备本地研究闭环、策略版本管理、评估与排行榜基础之后，做 Strategy Hub 是合理的下一阶段扩展。
- 但真正可用的 Hub 不是“加一个上传按钮”就够了，核心在于可复现评测、身份体系、审核、排名规则、反作弊和研究资产沉淀。

## 1. 为什么值得做

如果项目目标只是“单机策略研究工具”，Hub 不是必须。

如果项目目标是：

- 建立策略研究社区
- 沉淀公开策略语料
- 形成可比较、可复现、可讨论的排行榜
- 为 Pattern Library / Meta-Learner 提供更大的经验池

那么 Strategy Hub 很值得做。

它的价值主要有 4 点：

### 1.1 增长飞轮

用户会因为“能分享策略、看排名、看进化轨迹”而持续回来，而不是只把项目当一次性命令行工具。

### 1.2 数据飞轮

公开策略、评测结果、失败案例、版本迭代，会反过来喂给：

- Pattern Library
- Experience Store
- Meta-Learner
- 后续推荐系统

### 1.3 透明度飞轮

相比只晒收益曲线，AlphaEvo 的强项是可以晒：

- DSL
- 版本树
- 每轮修改原因
- benchmark 对比
- anti-overfit 结果

这比单纯“谁收益高”更适合做社区内容。

### 1.4 开源传播价值

一个可浏览、可点赞、可复现、可 fork 的策略 Hub，会比单纯 README 更容易形成传播。

## 2. 当前仓库已经具备的基础

当前仓库已经有一些适合做 Hub 的底座：

- 策略 YAML DSL，可读、可执行、可版本化
- `StrategyStore`，已能存储策略与评估结果
- `leaderboard`，已能做本地排行榜
- `EvaluationReport`，已有统一评估结构
- `ResearchLogger` / `Reporter`，已能沉淀运行记录和报告
- 进化树与 family/version 机制，天然适合做“fork / lineage”展示

这说明 Hub 并不是从零开始。

## 3. 当前还缺什么

如果要做公共 Strategy Hub，当前还缺少至少 6 类能力。

### 3.1 身份与权限

需要有：

- 用户身份
- 作者主页
- 策略所有权
- 策略 fork 关系
- 私有 / 公开可见性

当前本地 SQLite store 没有这些概念。

### 3.2 可复现评测协议

这是最关键的一项。

如果没有统一评测协议，Hub 会迅速失真。因为用户可以：

- 换不同数据源
- 换不同回测窗口
- 用不同滑点 / 手续费
- 只挑对自己有利的时间段

所以必须定义 canonical evaluation：

- 固定市场
- 固定数据源或数据快照版本
- 固定回测窗口
- 固定手续费 / 滑点
- 固定最小信号数门槛
- 固定 anti-overfit 规则

### 3.3 排名与社交信号拆分

必须把“研究表现”和“社区热度”拆开。

至少分成：

- Canonical Rank: 只看官方复评后的策略质量
- Popular Rank: 点赞、收藏、讨论热度
- Trending: 最近增长快的策略
- New: 新发布策略

不能把点赞直接并入策略质量分，否则会污染研究指标。

### 3.4 审核与反作弊

需要控制：

- 重复策略刷榜
- 轻微改名重复提交
- 人工篡改评测结果
- 用非 canonical 数据窗口刷高胜率
- 垃圾策略与广告内容

至少要有：

- 去重机制
- 官方重跑
- 审核状态
- 举报 / 下架能力

### 3.5 工件存储

Hub 不只是存一份 YAML。

至少还要保存：

- 策略 DSL
- 描述文本
- canonical evaluation JSON
- 报告 Markdown
- 研究日志
- 可选图表与回测摘要

### 3.6 Web 层

最终要做成社区产品，就需要：

- Web API
- 前端列表页 / 详情页
- 策略版本树可视化
- 点赞 / 收藏 / 评论 / fork

这部分当前仓库还没有。

## 4. 建议怎么做

建议按 4 个阶段推进，而不是一次性做大而全。

### 阶段 A：先做“可提交、可复评”的最小 Hub

目标：

- 用户能提交策略
- 系统能重跑官方评测
- 页面能展示官方结果

MVP 最小字段建议：

- `strategy_id`
- `family_id`
- `author_id`
- `title`
- `description`
- `market`
- `category`
- `tags`
- `yaml_content`
- `source_type`（原创 / fork / 官方模板）
- `submission_status`
- `canonical_run_status`
- `canonical_score`
- `canonical_report_url`
- `created_at`

这一阶段先不要急着上评论系统。

### 阶段 B：补社交层

新增：

- 点赞
- 收藏
- fork
- 评论
- 作者页
- 排行榜筛选

但注意：

- 点赞数不应直接影响 canonical score
- 只能影响热门排序和推荐流

### 阶段 C：补研究资产沉淀

新增：

- 每轮进化日志展示
- 策略 lineage 树
- 与 benchmark 的对比页
- 失败案例展示
- 参数敏感度、过拟合告警可视化

这一阶段会真正体现 AlphaEvo 与普通“策略榜单网站”的差异。

### 阶段 D：再考虑社区协作飞轮

例如：

- fork 后继续进化
- 公开挑战赛
- 官方评测赛道
- 不同市场 / 风格榜单
- Pattern Library 从公共策略中抽取高质量模式

## 5. 需要哪些准备

### 5.1 产品准备

先明确：

- Hub 是“研究展示平台”还是“策略比赛平台”
- 是否允许用户上传回测结果，还是必须官方重跑
- 是否允许私有策略
- 是否允许 fork 他人策略并再次上榜

这些规则会直接影响表结构和权限模型。

### 5.2 数据准备

需要统一：

- canonical 数据源
- canonical 时间窗口
- benchmark 选择
- 手续费 / 滑点 / 最小样本门槛
- 事件型策略是否单列榜单

否则排行榜会不可比较。

### 5.3 工程准备

建议新增的核心表：

- `users`
- `strategy_submissions`
- `canonical_runs`
- `strategy_votes`
- `strategy_bookmarks`
- `strategy_forks`
- `comments`
- `moderation_events`

还需要：

- 工件存储（本地对象存储 / S3 兼容）
- 队列或后台 worker 做复评
- 对外 API
- Web 前端

### 5.4 安全与合规准备

需要明确：

- 免责声明展示策略
- 禁止承诺收益
- 敏感信息与作者隐私处理
- 内容审核与垃圾提交治理

如果要公开运营，这部分不能后补。

## 6. 建议的实施顺序

如果现在开始做，建议顺序如下：

1. 先固化 canonical evaluation 规则。
2. 在现有 SQLite schema 之上抽象出“本地 store”与“Hub store”边界。
3. 先做只读 Web 展示页和只写提交接口。
4. 再做官方复评 worker。
5. 最后再上点赞、评论、热门流。

## 7. 对问题 3 的直接回答

如果问题是：

- “有没有必要做？”

答案是：有，但它是下一阶段的产品化扩展，不是当前研究闭环是否成立的前提。

如果问题是：

- “现在能不能直接做？”

答案是：能开始设计，但在真正公开前，必须先补齐复评协议、身份、审核、排名拆分和反作弊。
